"Post-training" covers every way you adapt a base model after pretraining. Pioneer offers three algorithms, all created through the same [`POST /felix/training-jobs`](/api-reference/training-jobs) endpoint and selected with the **`training_algorithm`** parameter:

| Algorithm | `training_algorithm` | What it optimizes | Dataset signal |
| --- | --- | --- | --- |
| **Supervised fine-tuning** | `sft` *(default)* | Imitate the assistant turns in your examples | Chat `messages` |
| **GRPO** | `grpo` | Maximize a reward over sampled completions | `prompt` + `answer` (+ a reward function) |
| **DPO** | `dpo` | Prefer chosen responses over rejected ones | `prompt` + `chosen` + `rejected` |

Omitting `training_algorithm` is equivalent to `sft`, so existing requests keep working unchanged.

<Note>
  All three algorithms are **LoRA-based**. A completed job produces a low-rank adapter that is hot-swapped onto the shared base model at serve time and exposed behind the same inference endpoints as base models — reference the training job's `id` as the `model_id` at inference time.
</Note>

## Choosing an algorithm

<CardGroup cols={3}>
  <Card title="SFT" icon="graduation-cap">

  </Card>
  <Card title="GRPO" icon="trophy">
    "Good" is a programmatic check — exact answers, numeric correctness, JSON validity, a rubric. The model explores and is reinforced toward higher reward.

  <Card title="DPO" icon="scale-balanced">

## Supervised fine-tuning (SFT)

SFT is the simplest path: provide a dataset of chat `messages` and the model learns to reproduce the assistant turns. See the [LLM fine-tuning guide](/guides/fine-tune-llm) for the full end-to-end walkthrough.

### LoRA, not full fine-tuning

Decoder SFT on Pioneer is **parameter-efficient (LoRA)**. The base model's weights stay frozen and training only learns a small set of low-rank adapter matrices, which keeps runs fast and cheap and produces a portable adapter rather than a full model copy. `training_type` defaults to `"lora"`, and for decoder LLMs that is the only supported value — full fine-tuning is reserved for GLiNER encoder models.

The LoRA capacity and core optimization settings are configurable; the defaults are sensible starting points:

| Field | Default | Purpose |
| --- | --- | --- |
| `lora_r` | `16` | LoRA rank — adapter capacity. Raise it for harder tasks or larger datasets. |
| `lora_alpha` | `32` | LoRA scaling factor (typically ~2× `lora_r`). |
| `lora_dropout` | `0.1` | Dropout applied to the adapter during training. |
| `learning_rate` | `2e-5` | Peak AdamW learning rate. |
| `batch_size` | `4` | Per-step batch size. |
| `nr_epochs` | `100` | Epoch ceiling; early stopping usually halts well before this. |
| `validation_data_percentage` | `0.2` | Fraction of the dataset held out for validation. |

```bash
curl -X POST https://api.pioneer.ai/felix/training-jobs \
  -H "X-API-Key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model_name": "my-sft-model",
    "base_model": "Qwen/Qwen3-8B",
    "training_type": "lora",
    "datasets": [{"name": "my-dataset", "version": "1"}],
    "lora_r": 16,
    "lora_alpha": 32,
    "learning_rate": 2e-5,
    "nr_epochs": 3
  }'
```

The dataset must contain a `messages` column where each row is a list of `{"role", "content"}` turns.

## GRPO

GRPO (Group Relative Policy Optimization) samples multiple completions per prompt and reinforces the ones that score highest against a **reward function**. Use it when you can express "good" as a programmatic check — for example, exact answers, numeric correctness, or JSON validity.

```bash
curl -X POST https://api.pioneer.ai/felix/training-jobs \
  -H "X-API-Key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model_name": "my-grpo-model",
    "base_model": "Qwen/Qwen3-4B-Instruct-2507",
    "training_type": "lora",
    "training_algorithm": "grpo",
    "datasets": [{"name": "gsm8k-grpo", "version": "1"}],
    "rl_config": {
      "reward_type": "numeric_match",
      "max_steps": 100,
      "group_size": 8
    }
  }'
```

**Dataset format.** Each row needs a `prompt` column (the policy input) and an `answer` column (the per-row ground truth read by the reward function).

**Reward functions** (`rl_config.reward_type`, required for GRPO):

| `reward_type` | Scores a completion as correct when… |
| --- | --- |
| `exact_match` | the normalized completion equals `answer` |
| `contains_substring` | `answer` appears anywhere in the completion |
| `numeric_match` | the extracted number matches `answer` (handles `#### 42`, `\boxed{42}`, "the answer is 42") |
| `choice_match` | the final multiple-choice letter matches `answer` |
| `regex_match` | the completion matches the supplied pattern |
| `json_match` | the parsed JSON deep-equals `answer` |
| `json_loose_match` | the parsed JSON loosely matches `answer` |
| `rougeL_match` | ROUGE-L against the `answer` reference(s) |
| `llm_as_judge` | a judge model scores the completion against a rubric |

<Note>
  When `reward_type` is `llm_as_judge`, Pioneer mints and manages the judge credential for you — you never supply an API key. Optional judge knobs include `llm_judge_model`, `llm_judge_rubric`, and `llm_judge_score_scale`.
</Note>

## DPO

DPO (Direct Preference Optimization) learns from preference pairs — for each prompt you supply a better (`chosen`) and worse (`rejected`) response.

```bash
curl -X POST https://api.pioneer.ai/felix/training-jobs \
  -H "X-API-Key: YOUR_API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model_name": "my-dpo-model",
    "base_model": "meta-llama/Llama-3.1-8B-Instruct",
    "training_type": "lora",
    "training_algorithm": "dpo",
    "datasets": [{"name": "my-preferences", "version": "1"}],
    "rl_config": {
      "dpo_beta": 0.1
    }
  }'
```

**Dataset format.** Each row needs `chosen` and `rejected` responses, and (optionally) a shared `prompt`. `rl_config` itself is required for DPO, but every key inside it is optional — server defaults are applied.

## Monitoring and deploying

Poll `GET /felix/training-jobs/:id` for status. The job advances through `complete` (GPU training finished) to `deployed` — the terminal success state, reached once the resulting adapter is deployed automatically. When it reaches `deployed`, the model is available for [inference](/concepts/inference): pass the job `id` as the `model_id`.

```bash
curl https://api.pioneer.ai/felix/training-jobs/YOUR_JOB_ID \
  -H "X-API-Key: YOUR_API_KEY"
```
